## 一、技术可行性分析 ✅

### 1.1 现有基础设施支持

**ChromaDB 原生支持的数据：**
```python
results = collection.query(
    query_texts=["查询"],
    n_results=5,
    include=['documents', 'metadatas', 'distances', 'embeddings']  # ✅ 全部可获取
)
```

**已有的数据结构：**
- ✅ `distances`: 相似度分数（可转换为百分比）
- ✅ `metadatas`: 包含 `uri`, `filename`, `l0_abstract`, `l1_overview` 等
- ✅ `documents`: 原始文档内容
- ✅ `embeddings`: 向量表示（可选获取）

### 1.2 可实现的功能

| 功能 | 实现方式 | 难度 |
|------|---------|------|
| **匹配度百分比** | `(1 - distance) * 100%` 或归一化 | ⭐ 简单 |
| **命中位置** | 从 metadata 提取章节信息 | ⭐⭐ 中等 |
| **关键词高亮** | 使用 rerank 已有的关键词提取 | ⭐ 简单 |
| **检索路径** | 记录 Query → Embedding → Results | ⭐⭐ 中等 |
| **相关性解释** | 结合语义分数 + 关键词匹配 | ⭐⭐⭐ 较难 |

## 二、实现难度评估 ⭐⭐⭐☆☆

### 2.1 需要改造的模块

**简单改造（1-2小时）：**
1. ✅ `cbridge.py` - 增强搜索结果展示
2. ✅ `core/i18n.py` - 添加新的翻译文本

**中等改造（3-5小时）：**
3. ⚠️ `core/runtimes/qmd_runtime.py` - 增强返回数据
4. ⚠️ `core/managers/openviking_manager.py` - 添加元数据
5. ⚠️ `cbridge.py` - 实现关键词高亮和路径追踪

**复杂改造（可选，5-10小时）：**
6. 🔧 文档解析时提取章节结构（需要改造 parser）
7. 🔧 可视化展示（需要 Rich 库的高级功能）

### 2.2 技术挑战

**挑战 1：章节位置提取** ⚠️
- **问题**：当前解析器只返回纯文本，没有结构化信息
- **解决方案**：
  - 简单版：基于文本模式匹配（如 "第X章"、"Chapter X"）
  - 完整版：使用 Docling 的结构化解析能力提取目录

**挑战 2：距离分数归一化** ⚠️
- **问题**：ChromaDB 默认返回 L2 距离，不是 0-1 的相似度
- **解决方案**：
  ```python
  # L2 距离转相似度
  similarity = 1 / (1 + distance)
  # 或使用 cosine 距离（需配置）
  similarity = 1 - cosine_distance
  ```

**挑战 3：关键词匹配可视化** ✅
- **问题**：需要在文本中高亮关键词
- **解决方案**：已有 `_rerank_by_keywords` 函数，可复用

### 4.2 技术架构建议

```python
# 新增数据结构
class ExplainableResult:
    uri: str
    content: str
    score: float
    explanation: {
        'semantic_score': float,      # 语义分数
        'keyword_score': float,       # 关键词分数
        'matched_keywords': list,     # 匹配的关键词
        'location': {                 # 位置信息

            'section': str,
            'page': int
        },
        'retrieval_path': list,       # 检索路径

    }
```

## 六、最终建议 ✨

### ✅ 强烈推荐实现

**理由：**
1. **技术成熟** - ChromaDB 原生支持所需数据
