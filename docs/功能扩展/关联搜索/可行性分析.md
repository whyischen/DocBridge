## 核心问题拆解

多跳推理的本质是：**从问题 A → 中间信息 B → 最终答案 C**

关键挑战：
1. **理解用户真实意图**（"超支的原因"需要找因果关系）
2. **判断当前信息是否足够**（何时停止检索）
3. **规划下一步检索方向**（应该查什么）
4. **综合多个片段生成答案**（如何组织信息）

---

| 能力维度 | 无 LLM 方案 | 可行性 | 质量评分 |

## 详细分析

### 1️⃣ 意图理解（部分可行，质量低）

**用户问："Q3 营销预算超支的原因是什么？"**

**无 LLM 能做什么：**
```
✅ 提取关键词：Q3、营销、预算、超支、原因
✅ 识别实体类型：时间(Q3)、部门(营销)、财务(预算)
✅ 识别问题类型：因果关系（"原因"）
```

**无 LLM 做不到什么：**
```
❌ 理解"超支"隐含需要对比"预算 vs 实际"
❌ 理解"原因"需要找"支出明细"和"外部因素"
❌ 理解问题的深层语义（用户真正想知道什么）
```

**结论**：只能做表面的关键词匹配，无法理解深层意图。

---

### 2️⃣ 停止判断（可行，质量中等）

**问题**：检索到什么程度可以停止？

**无 LLM 方案：**
```python
# 规则 1：相似度阈值
if avg_similarity > 0.75:
    stop()  # 结果质量高，停止

# 规则 2：结果数量
if total_results > 15:
    stop()  # 已经找到足够多的片段

# 规则 3：新增结果递减
if new_results_count < 2:
    stop()  # 后续检索收益低

# 规则 4：实体覆盖度
if extracted_entities_covered > 0.8:
    stop()  # 已覆盖大部分实体
```

**优点**：这些规则简单有效，可以避免无限检索
**缺点**：无法判断"信息是否真正回答了问题"

**结论**：可以用规则控制检索深度，但无法保证答案完整性。

---

### 3️⃣ 路径规划（有限可行，质量低）

**问题**：下一跳应该查什么？

#### 方案 A：预定义规则树（死板但可用）

```
IF 问题类型 == "原因分析":
    第一跳：检索主题本身（"Q3 营销预算"）
    第二跳：检索明细数据（"Q3 营销支出明细"）
    第三跳：检索外部因素（"市场变化" + "竞争对手"）

IF 问题类型 == "对比分析":
    第一跳：检索对象 A
    第二跳：检索对象 B
    第三跳：检索对比维度

IF 问题类型 == "趋势分析":
    第一跳：检索当前时间点
    第二跳：检索历史时间点
    第三跳：检索行业基准
```

**优点**：对常见问题类型有效
**缺点**：无法处理规则外的问题

#### 方案 B：实体链式扩展（灵活但浅层）

```
第一跳：检索 "Q3 营销预算超支"
  ↓ 提取实体
  发现：Q3、营销、预算、超支、抖音（从结果中提取）

第二跳：检索 "抖音 Q3 营销"
  ↓ 提取新实体
  发现：CPM、投放、ROI

第三跳：检索 "抖音 CPM 上涨"
  ↓ 返回结果
```

**优点**：自动发现关联实体，不需要预定义
**缺点**：只能做"词汇关联"，无法做"逻辑推理"

**结论**：可以做浅层的实体链接，但无法做深层的因果推理。

### 4️⃣ 答案生成（几乎不可行）

**问题**：如何把 3 跳的 15 个片段组织成连贯的答案？

**无 LLM 方案：**
```python
# 方案 1：简单拼接（质量极差）
answer = "\n\n".join([r['content'] for r in results])

# 方案 2：模板填充（只适用于特定问题）
template = """
Q3 营销预算情况：{hop1_result}
具体支出明细：{hop2_result}
超支原因分析：{hop3_result}
"""

# 方案 3：抽取式摘要（需要额外模型）
from sumy.summarizers.lsa import LsaSummarizer
summary = summarizer(combined_text, sentences_count=5)
```

**致命问题**：
- 无法理解片段之间的逻辑关系
- 无法过滤无关信息
- 无法用自然语言表达因果关系
- 无法做推理（"CPM 上涨 → 导致超支"）

**结论**：答案生成几乎必须依赖 LLM。

## 真实能力边界

### ✅ 无 LLM 能做到的：

1. **关键词网络扩展**
   - 用户查 "营销预算" → 自动扩展查 "营销支出"、"营销费用"
   - 效果：召回率提升 30-50%

2. **实体链式检索**
   - 从第一跳结果中提取 "抖音" → 第二跳查 "抖音投放"
   - 效果：能找到关联文档，但不保证相关性

3. **时间序列检索**
   - 用户查 "Q3" → 自动查 "Q2"、"Q4"（对比）
   - 效果：适合趋势分析类问题

4. **目录递归检索**
   - 发现 "docs/marketing/" → 自动检索该目录下所有文件
   - 效果：适合结构化文档库

### ❌ 无 LLM 做不到的：

1. **因果推理**
   - 无法理解 "CPM 上涨" 是 "超支" 的原因
   - 只能把两个片段都返回，让用户自己判断

2. **信息综合**
   - 无法把 3 个片段的信息合并成一句话
   - 只能原样返回所有片段

3. **动态规划**
   - 无法根据中间结果调整检索策略
   - 只能按预定义规则执行

4. **答案生成**
   - 无法生成 "超支主因是抖音 CPM 从¥15 涨到¥45"
   - 只能返回包含这些数字的原始片段

## 实用价值评估

### 场景 1：简单事实查询（✅ 有价值）

**问题**："Q3 营销预算是多少？"

- 第一跳：检索 "Q3 营销预算" → 直接找到答案
- **无需多跳**，传统 RAG 就够了

### 场景 2：关联信息查询（✅ 有价值）

**问题**："抖音投放的所有相关文档"

- 第一跳：检索 "抖音投放"
- 第二跳：从结果中提取 "Q3"、"CPM" → 检索 "抖音 Q3"、"抖音 CPM"
- 第三跳：检索 "抖音复盘报告"

**效果**：能找到更全面的相关文档，召回率提升明显

### 场景 3：因果分析查询（❌ 价值有限）

**问题**："Q3 营销预算超支的原因是什么？"

- 无 LLM 方案：返回 15 个片段，包含 "预算表"、"支出明细"、"CPM 数据"
- **用户仍需自己阅读并推理**
- 相比传统 RAG：多返回了一些关联片段，但没有给出答案

**效果**：比单跳好，但远不如 LLM 推理

## 最终结论

### 可行性评分：6/10

**能实现的核心价值：**
- ✅ 提升召回率（从 5 个片段 → 15 个片段）
- ✅ 发现关联文档（自动扩展检索范围）
- ✅ 减少用户手动搜索次数

**无法实现的核心价值：**
- ❌ 真正的"推理"（因果、对比、归纳）
- ❌ 自然语言答案生成
- ❌ 动态策略调整

### 推荐方案：混合架构

```
用户查询
  ↓
[无 LLM] 多跳检索（扩展召回）
  ↓ 返回 15 个片段
[可选 LLM] 综合推理（生成答案）
  ↓
返回给用户
```

**优势**：
- 默认模式：无 LLM，只返回扩展的检索结果（召回率高）
- 高级模式：用户可选接入本地 LLM（Ollama/LM Studio）做最后的答案生成
- 完全本地化：即使用 LLM，也是本地模型

**这样既保持了 ContextBridge 的"100% 本地隐私"特性，又给高级用户提供了推理能力的选项。**
